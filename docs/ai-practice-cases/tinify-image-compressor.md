# AI 实践案例说明｜Tinify 图片批量压缩 Skill

> 申报定位建议：团队方法复用。若暂时无法提供其他成员的真实复用记录，应改选“岗位场景应用”，不要仅凭已经创建 Skill 就申报团队复用。

## 1. 案例名称

使用 AI 沉淀 Tinify 图片批量优化 Skill，解决前端项目和活动页面图片资源体积过大、加载缓慢及运行内存占用过高的问题。

## 2. 真实场景与原问题

- **适用岗位/团队：** 前端研发、视觉设计、活动运营、内容运营及需要持续交付官网、专题页、营销活动页的团队。
- **具体任务：** 前端项目和活动页面通常包含 Banner、背景图、商品图、人物图、装饰图及移动端适配图等大量图片资源。通过统一 Skill 批量扫描 PNG、JPEG、WebP 文件，在不改变业务内容和视觉表现的前提下进行压缩，并根据页面实际展示尺寸完成缩放、AVIF/WebP/JPEG/PNG 格式转换、透明背景处理和必要的元数据保留。
- **原来的问题：**
  - 设计交付的图片通常保持较高分辨率和质量，直接放入项目会显著增加静态资源体积、构建产物和发布包大小。
  - 页面首次访问时需要下载大量图片，消耗更多网络带宽，弱网和移动端环境下容易出现首屏加载慢、图片长时间空白或活动内容延迟展示。
  - 图片实际展示区域较小，但源文件尺寸远高于展示尺寸；浏览器解码后仍会按照像素尺寸占用内存，图片数量较多时会增加页面运行内存和渲染压力，严重时可能出现卡顿或页面被系统回收。
  - 人工逐张压缩、改尺寸和转换格式效率低，容易遗漏；不同人员的处理标准也不一致，无法保证所有活动资源都经过优化。
  - 直接覆盖原图存在不可逆风险，批量处理多个目录时也容易破坏相对路径，增加前端替换和回归检查成本。
  - Tinify 正式压缩会消耗 API 配额，如果不先确认文件范围、处理参数和输出路径，可能产生不必要的调用。
- **使用频率：** 高频、固定流程；适用于每次活动上线、版本发布、官网性能优化和存量图片治理。实际月均使用次数、图片数量和原始资源体积为 **[待补充真实数据]**。

## 3. AI 介入方式

- **使用工具：** Codex、Node.js、Tinify Node.js API、项目文件系统。
- **输入材料：** 本地图片文件或目录、期望输出目录、尺寸要求、目标格式、透明背景颜色、元数据保留要求及并发数量。
- **核心 Prompt / 模板：**

  ```text
  请使用 tinify-image-compressor 扫描前端项目或活动页的图片目录，先通过 dry-run
  输出文件数量、原始资源体积、输入输出路径映射和计划执行的转换，不要覆盖原图，
  也不要消耗 Tinify 配额。经我确认后，再将图片压缩到独立输出目录，并根据页面
  实际展示需求完成尺寸调整或 WebP/AVIF 格式转换。最后汇总成功、跳过和失败数量，
  对比压缩前后的总体积，并标明可替换到项目中的输出位置。
  不得输出或写入真实 Tinify API Key。
  ```

- **AI 负责的环节：** 检查输入文件、识别支持的图片类型、规划安全输出路径、生成 Dry Run、校验转换和尺寸参数、调用批处理 CLI、分类 Tinify 错误并汇总处理结果。
- **人工判断/修改/复核的环节：** 确认哪些资源需要处理、选择清晰度和体积之间的平衡、决定目标尺寸与格式、配置和管理 API Key、确认配额消耗、抽查图片视觉质量并决定是否替换项目资源。
- **人工判断的复杂环节：** 透明 PNG 转 JPEG 的背景颜色、裁切方式、版权/创建/位置信息是否保留，以及是否允许覆盖原图等决策必须由人工确认。

标准调用方式：

```bash
node scripts/compress-images.mjs assets --out dist/assets-optimized --dry-run
node scripts/compress-images.mjs assets --out dist/assets-optimized
node scripts/compress-images.mjs hero.png --out dist/hero.webp --convert webp
node scripts/compress-images.mjs poster.jpg --out dist/poster.jpg --resize fit:1600x1600
```

正式执行前必须提醒使用者：压缩会调用 Tinify API 并消耗账号配额。默认输出到独立目录，不覆盖原始图片。

## 5. 产出与效果

- **最终产出：**
  - 可复用的 Tinify 图片压缩 Skill。
  - Node.js 批处理 CLI，支持图片发现、压缩、尺寸调整、格式转换、透明背景填充、元数据保留和并发限制。
  - Dry Run 路径预演及 API 配额保护机制。
  - 原图保护和相对目录结构保留规则。
  - 文件处理数量、输出位置、压缩前后体积及异常信息的统一报告。
- **节省时间/提升质量：** 将逐张处理图片改为一次批量任务，通过统一参数完成压缩、按展示尺寸缩放和现代格式转换，降低静态资源体积、网络传输量及超大图片解码后的内存压力，同时减少漏处理、路径错误和覆盖原图的风险。
- **性能效果：** 图片文件体积降低后，可减少页面下载流量并改善首屏及活动内容加载速度；将远大于展示区域的图片缩放到合理像素尺寸后，可进一步减少浏览器解码内存和渲染压力。最终效果应以构建产物、Network、Lighthouse 或浏览器性能数据验证。
