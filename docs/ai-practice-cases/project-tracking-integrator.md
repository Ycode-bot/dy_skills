# AI 实践案例说明｜项目埋点集成 Skill

> 申报定位建议：团队方法复用。若暂时没有团队成员实际复用证据，可先按“岗位场景应用”申报。

## 1. 案例名称

使用 AI 沉淀项目埋点集成 Skill，解决多平台埋点需求理解、代码接入、重复上报和验收不一致问题。

## 2. 真实场景与原问题

- **适用岗位/团队：** 前端研发、全栈研发、产品经理、数据分析、增长及需要接入神策、GA4、GTM、Google Ads 等分析系统的团队。
- **具体任务：** 接收产品或数据团队提供的文字、表格、截图和文档形式埋点需求，扫描项目既有埋点架构，将需求整理为事件契约，并完成或输出事件代码、调用位置、平台路由、文档同步和验证方案。
- **原来的问题：**
  - 需求来源格式不统一，触发条件、字段来源、平台归属、上报时机和验收方式经常缺失。
  - 项目可能同时存在神策、GA4、GTM 和 Google Ads，容易因为“SDK 已安装”而错误地默认双报。
  - 直接在页面中调用 SDK 会绕过项目公共封装，导致命名、公共字段、身份和异常处理不一致。
  - 曝光、异步结果、关闭弹窗等事件容易因为重新渲染、重复回调或自动关闭而重复上报。
  - 代码、埋点地图、事件目录和数据团队文档可能不同步，研发完成后仍难以验收。
- **使用频率：** 高频/固定流程；适用于每次新增、修改、审计或验收埋点需求。实际月均需求次数为 **[待补充真实数据]**。

## 3. AI 介入方式

- **使用工具：** Codex、代码搜索工具、项目现有测试/类型检查、浏览器 Network 或对应分析平台调试工具。
- **输入材料：** 目标代码仓库、项目说明、依赖和框架配置、埋点 SDK 初始化、公共上报方法、现有业务封装、事件目录、埋点表格/截图、产品需求和验收口径。
- **核心 Prompt / 模板：**

  ```text
  请先扫描项目现有埋点架构，不要假设框架、SDK、事件名称或目录结构。
  将每个需求规范化为事件契约：触发条件、目标平台、事件名称、业务封装、
  属性名称/类型/来源、上报时机、代码位置、去重规则和验证方式。
  复用现有公共上报层，不要在 UI 中直接调用 SDK，也不要因为同时安装多个平台就默认双报。
  埋点完成后，使用同一事件契约核对浏览器实际上报和神策入库数据，
  区分未触发、未发送、未入库、重复上报、字段不一致和查询失败。
  最终输出 Scan Findings、Implementation Plan、Patch Draft、Verification Checklist 和脱敏验收报告。
  ```

- **AI 负责的环节：** 扫描依赖和初始化逻辑、追踪公共封装与调用位置、识别事件命名和公共字段、把非结构化需求转成事件契约、发现歧义和重复风险、设计代码修改草案、生成验证清单，并通过神策只读查询 API 将入库事件与数据文档逐字段对账。
- **人工判断/修改/复核的环节：** 明确业务问题、确认最终事件口径和字段含义、决定每个平台是否必须上报、确认隐私与用户授权规则、审核代码、在测试环境验证实际请求并由数据团队验收。
- **人工判断的复杂环节：** 平台目标为 `unknown`、曝光定义不明确、支付或转化事件存在重试、多个 SDK 可能重复触发、字段涉及个人信息或业务口径冲突时，必须由产品、数据和研发共同确认，AI 不得自行猜测。

标准事件契约包括：

| 字段 | 内容 |
|---|---|
| Trigger | 精确的用户动作或业务状态变化 |
| Targets | 各平台为 required、optional、disabled 或 unknown |
| Event | 各平台实际事件名称 |
| Wrapper | 项目业务封装函数 |
| Properties | 字段名、类型、必填性、枚举、映射和来源 |
| Timing | 操作前后、一次或可重复 |
| Location | 页面、组件、Composable 或 Service |
| Deduplication | 防止重渲染、重试或重复回调 |
| Validation | Network、DebugView、GTM Preview 或 SDK Debug 验证方式 |

## 4. 本人贡献与参与程度

- **本人角色：** 建议按真实情况填写“发起人 / 主导者 / 主要执行者 / 协作者 / 使用者”。
- **本人具体完成的工作：**
  - 总结项目埋点扫描范围和成熟度分类方法。
  - 将表格、截图和文字需求统一为可核对的事件契约。
  - 定义多平台路由规则，避免根据 SDK 安装情况盲目双报。
  - 明确业务事件封装、触发位置、去重、隐私和验证要求。
  - 整理 ImaStudio 场景下的神策与 GA/GTM 既有架构及文档同步规则。
  - 维护 Skill、参考资料和后续反馈迭代。
- **本人负责的关键判断、设计或推进环节：** 确定“先扫描、再设计”的基本方法；建立 `required/optional/disabled/unknown` 平台状态；规定一个业务触发点分别映射各平台独立载荷；把文档同步和运行时验证纳入完成标准。
- **AI 负责的部分：** 大范围检索代码、沿导入关系追踪调用、归纳既有约定、生成事件契约和 Patch 草案、检查重复路径、整理验证矩阵。

## 5. 产出与效果

- **最终产出：**
  - 可复用的项目埋点集成 Skill。
  - 项目埋点架构扫描清单和成熟度分类方法。
  - 标准事件契约模板。
  - 神策、GA4、GTM、Google Ads 多平台路由规范。
  - ImaStudio 项目模式参考。
  - 代码设计、文档同步、去重及运行时验证清单。
  - 神策事件验收 CLI，支持离线事件对账、SQL Query API 入库验证、重复检测、字段类型/枚举/取值校验和脱敏 Markdown/JSON 报告。
- **节省时间/提升质量：** Skill 把需求理解、代码检索、事件设计、文档同步和验收组织成固定流程，重点减少漏字段、错平台、重复上报和代码文档不一致。请补充真实数据，例如：单个埋点需求分析由 **[原耗时]** 降至 **[现耗时]**；一次验收发现问题由 **[数量]** 降至 **[数量]**。
