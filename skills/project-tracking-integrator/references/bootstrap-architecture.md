# 从零建立埋点体系

仅当扫描结果为 `absent` 且用户明确要求实施时使用本指南。目标是建立可验证的最小基础设施，不是一次性给所有页面加事件。

## 实施前决策

先确认以下信息；缺少会改变架构的项目必须标记为阻塞项：

| 决策 | 必须确认的内容 |
|---|---|
| 平台 | 神策、GA4、GTM、Google Ads 或其他目标；不要自行选择 |
| 环境 | 开发、测试、生产项目和开关策略 |
| 配置 | 客户端采集地址或 Measurement ID；查询凭证必须保持服务端/本地私有 |
| 隐私 | 同意前是否允许初始化或发送、退出授权如何处理 |
| 身份 | 匿名 ID、登录关联、跨设备、退出重置规则 |
| 公共属性 | 应用版本、渠道、页面、实验等经过批准的稳定字段 |
| 部署 | CSR、SSR、服务端事件、容器或 Tag Manager 边界 |

## 最小架构

按项目框架和模块规范实现以下层次：

```text
application bootstrap
  └─ consent/environment gate
      └─ vendor SDK initialization
          ├─ identity lifecycle
          ├─ approved common properties
          └─ vendor transport
              └─ typed business wrappers
                  └─ business trigger owners
```

每层职责：

1. **配置层**：读取非敏感的客户端配置；区分环境；默认关闭无效配置。
2. **初始化层**：只在客户端或允许的运行环境初始化一次；处理重复挂载和 SSR。
3. **授权层**：遵守项目的隐私同意、撤回和退出机制。
4. **身份层**：定义匿名、登录关联和退出重置，不复用访问令牌或原始个人信息。
5. **传输层**：封装厂商 SDK；做可用性保护；不得让埋点异常中断业务。
6. **业务层**：暴露有类型、语义稳定的 `trackXxx` 方法；厂商字段不泄漏到所有页面。
7. **触发层**：在真实业务状态变化处调用；负责一次性、重试和去重语义。

## 多平台原则

- 为每个平台保留独立初始化、身份映射和 payload。
- GTM 负责路由时，避免应用同时用 `gtag` 发往同一目的地。
- 一个业务触发可以调用多个目标 wrapper，但不能把厂商 schema 合并成最低公分母。
- 对购买等转换使用平台支持的稳定交易标识，不能用时间戳冒充业务 ID。

## 第一个代表事件

只接入一个有明确触发和验证方式的事件，用它证明：

1. SDK 在正确环境初始化。
2. 授权和身份符合预期。
3. 公共属性与业务属性正确。
4. wrapper 可达且不影响业务。
5. 浏览器或 SDK 能观察到发送。
6. 数据平台能查询到入库数据，或明确记录当前平台无法做 API 入库验证。

基础事件通过后，再批量实施其余契约。

## 完成标准

- 没有硬编码密钥或生产查询凭证。
- 不存在重复初始化和明显的直接 SDK 散落调用。
- 登录、退出和匿名状态经过测试。
- 至少一个代表事件取得源码、发送和入库/平台诊断证据。
- 契约、事件字典、代码和验收报告保持一致。
